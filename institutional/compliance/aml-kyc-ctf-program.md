# AML / KYC / CTF Compliance Program v1.0

[VOLKOV INTELLIGENCE SYSTEMS] CLRTY AML / KYC / CTF COMPLIANCE PROGRAM Version: 1.0 Applies to: $CLRTY Token Ecosystem, MIRRA / Σ YMBRA Infrastructure, Treasury, Validators, and Associated Services Entity: Volkov Intelligence Systems / Clarity Fintech 1.

POLICY OBJECTIVE The purpose of this AML/KYC/CTF Policy is to: ● Prevent use of the $CLRTY ecosystem for money laundering, terrorist financing, sanctions evasion, or illicit financial activity ● Ensure compliance with: ○ FATF Recommendations ○ FinCEN (U.S.) ○ OFAC Sanctions Programs ○ EU AMLD5/6 ○ UK MLR 2017 ● Establish risk-based identity verification, transaction monitoring, and governance controls ● Align with the protocol’s multi-layer governance and non-unilateral control architecture 2.

COMPLIANCE GOVERNANCE STRUCTURE 2.1 Oversight Model CLRTY adopts a multi-layer governance model , consistent with protocol design: ● Primary Governance Multisig (4-of-7) Controls: ○ Treasury ○ Protocol upgrades ○ Compliance parameter changes ● Emergency Guardians (3-of-5) Controls: ○ Pause / freeze mechanisms (incident-only scope) ● Time-lock Layer (24–72 hours) Ensures: ○ Public visibility before execution ○ Compliance review window 📌 This structure ensures: ● No unilateral compliance override ● Separation between operational and emergency powers ● Auditable governance actions 2.2 Compliance Roles Chief Compliance Officer (CCO) ● Owns AML/KYC program ● Reports to governance multisig ● Signs annual attestation Money Laundering Reporting Officer (MLRO) ● Handles suspicious activity reports (SARs) ● Liaison with regulators Sanctions Officer ● Oversees OFAC / EU / UN screening ● Maintains sanctions lists Risk Committee ● Reviews: ○ High-risk users ○ Governance-level risks ○ Cross-chain exposure 3.

RISK-BASED AML FRAMEWORK 3.1 Risk Categories CLRTY classifies users and entities into: Low Risk ● Retail users (non-custodial wallets) ● Small transaction volumes ● No cross-chain complexity Medium Risk ● Active traders ● Cross-chain participants ● API users / developers High Risk ● Institutional flows ● Liquidity providers ● Market makers ● Treasury counterparties ● Privacy-enhanced routing users 3.2 Risk Factors ● Jurisdiction (FATF high-risk countries) ● Transaction velocity & volume ● Wallet entropy / fragmentation ● Cross-chain bridge frequency ● Interaction with: ○ Mixers ○ Sanctioned addresses ○ Darknet-linked flows 4.

KYC (KNOW YOUR CUSTOMER) PROGRAM 4.1 Customer Identification Program (CIP) Required Information: Individuals: ● Full legal name ● Date of birth ● Government-issued ID ● Proof of address ● Liveness verification Entities: ● Legal name ● Registration documents ● Beneficial ownership (>25%) ● Director verification ● Business purpose 4.2 KYC Tiers Tier 0 – Permissionless ● Non-custodial wallet use ● No KYC required ● Monitoring only Tier 1 – Basic Access ● Email + wallet verification ● Limited transaction thresholds Tier 2 – Verified ● Full KYC ● Higher limits ● Access to staking, treasury programs Tier 3 – Institutional ● Enhanced Due Diligence (EDD) ● Source of funds verification ● Legal opinion / compliance docs 4.3 Enhanced Due Diligence (EDD) Triggered for: ● High-value transactions ● High-risk jurisdictions ● Institutional accounts Includes: ● Source of wealth verification ● Business model validation ● Ongoing monitoring 5.

SANCTIONS SCREENING PROGRAM 5.1 Screening Sources CLRTY screens against: ● OFAC SDN List ● EU Consolidated List ● UN Sanctions List ● UK HMT Sanctions List ● Commercial blockchain intelligence providers 5.2 Screening Procedures Onboarding Screening ● Identity screened before approval ● Wallet address screened against: ○ Known illicit clusters ○ Sanctioned entities Ongoing Screening ● Continuous monitoring of: ○ Wallet interactions ○ Counterparties ○ Smart contract exposure Transaction-Level Screening ● Real-time flagging of: ○ High-risk transfers ○ Sanctioned address interactions 5.3 Blocking & Freezing Controls If a sanctioned match is detected: ● Immediate: ○ Transaction halt ○ Wallet restriction ● Escalation: ○ Compliance review ● If confirmed: ○ Freeze (via governance / guardian path) ○ Report to authorities 📌 Emergency controls are strictly limited to pause/freeze only , preventing abuse of power 6.
