# Volume II — Security & Governance — Chapter 4

Chapter 6 — Burn Mechanism & Unspendable Address Verification CLRTY supports both event-driven and protocol-driven burn.

Both routes send tokens to a single canonical unspendable address (the "dead" address) whose non-recoverability is cryptographically demonstrable.

Verification of Unspendability The canonical burn address is a well-known address (e.g., 0x000…dEaD) whose corresponding private key has never been generated by any known keypair scheme; brute-force recovery is computationally infeasible under any known cryptographic assumption.

The FORTRESS dashboard publishes the ongoing balance at the burn address and cross-checks it against cumulative Transfer(to=burn) events.

Protocol fee-burn hooks (where enabled) route a defined fraction of transaction fees to the burn address atomically within the fee-collection function.

SECURITY NOTICE No superuser role — including Emergency Guardian — has the ability to withdraw from the burn address.

This is not a permission that has been removed; it is a permission that mathematically does not exist.

CLRTY Protocol — Volume II Security, Governance & Exchange Compliance Addendum --- Chapter 8 — Tax / Fee / Reflection Logic Where CLRTY or an ecosystem token implements a fee, reflection, or "swap-and-liquify" pattern, V olume II mandates that the entire swap-and-liquify path be atomic within a single transaction.

Partial-state failures — where fees are collected but the resulting swap or liquidity-add fails — are not tolerated.

Atomicity Requirements Fee collection, swap-to-base-asset, and liquidity-add (or treasury-forward) occur inside one external transaction.

All internal calls are guarded by a reentrancy modifier.

If any sub-step reverts, the whole transaction reverts; there is no path that leaves the fee-collector contract in a partially-drained state.

Fee routing is expressed as an on-chain routing tree whose leaves (treasury, LP, burn, staker rewards) sum to 100% and are governed by the primary multisig.

RISK Reflection and swap-and-liquify patterns have historically been a source of MEV extraction and sandwich attacks.

Where they are used, CLRTY additionally routes their triggering transactions through HELIX private execution to eliminate that vector.

• • • • CLRTY Protocol — Volume II Security, Governance & Exchange Compliance Addendum --- Chapter 9 — Oracle Reliability (HELIX + Chainlink) Any CLRTY component that reads an external price — HELIX allocation routing, liquidation logic, cross- agent settlement — must do so through an oracle abstraction that supports a primary feed, a fallback feed, staleness rejection, and deviation bounds.

Oracle Abstraction Primary feed: Chainlink-style push oracle where available; otherwise a signed off-chain feed with on- chain verifier.

Fallback feed: Independent secondary provider with a distinct signing set.

Heartbeat & deviation: A configured heartbeat per pair; any reading older than the heartbeat is rejected as stale.

Deviation guardrails: Reads that diverge from a short trailing mean beyond a configured band trigger a fallback swap or protocol pause.

Delay-tolerant logic: Consumer contracts explicitly define behavior when the oracle is unavailable: safe pause, last-known-good reading with expiry, or fallback path.
